# TheTattooCore Product Plan

## Core Experience

- Feed: image posts and short reel clips, 40-word captions, likes, short 40-word comments, comment likes, and comment replies. Comment likes/replies are live for launch on detail pages; the main 4U feed now shows only comment counts linking into the full post so the scroll stays fast. Original 4U authors must be able to edit or delete their own posts from the main card and detail page. Done for launch with owner-only main-card/detail controls and server checks that require an actual owned row.
- Threads: longer posts, optional image, replies, likes, comment likes, and community discussion. Comment likes/replies are live for launch on detail pages; the main Gossip feed now shows only comment counts linking into the full thread so the scroll stays fast. Original Gossip authors must be able to edit or delete their own posts from the main card and detail page, with moderation/audit rules preserved. Done for launch with owner-only main-card/detail controls and server checks that require an actual owned row.
- Comments: original authors must be able to edit/delete their own comments and replies. Post/thread owners should be able to hide/delete/block disruptive comments on their own posts, and all signed-in users should be able to report comments. These controls must appear wherever comments are rendered, including 4U/Gossip cards, detail pages, opened notifications, and deep-link contexts. Done for launch on home cards and 4U/Gossip detail pages, with compact styled controls on detail pages.
- Comment-delete RPCs currently use authenticated `SECURITY DEFINER` functions with `auth.uid()` guards so authors and post/thread owners can delete comments consistently. Revisit after launch to move the implementation into a private schema or service-role route if Supabase advisor noise needs to be fully eliminated.
- Marketplace/Stuff: verified artists, studios, and vendors can buy, sell, and trade approved body-art goods and services; fans can browse but not transact or contact sellers. Listing owners must be able to edit or delete their own Stuff listings while preserving admin review state, public safety filters, and transaction history. Done for launch with owner edit/archive controls on Stuff detail pages, main-card manage entry points, and server checks that require an actual owned row.
- Merch: a separate public-buyable marketplace for artist, studio, vendor, and TheTattooCore merchandise such as T-shirts, prints, art, stickers, aftercare-adjacent brand goods, and other safe fan-facing products. Merch should not be mixed with pro Stuff because the general public can buy Merch, while Stuff stays verification-gated. Product owners/admins must be able to edit or delete/archive Merch products; deletion should not break paid order history. Started for launch with owner archive controls, main-card manage entry points, seller edits that move active/approved products back to admin review before checkout opens again, seller archive through RLS instead of a service-key dependency, and server checks that require an actual owned row.
- Messages: integrated direct messenger for bookings, marketplace questions, and collaboration. Senders should be able to delete their own DM messages only while the other user has not read them; after read, use clear deletion/retention rules instead of silent removal. DM threads should show small delivered/read status icons under outgoing messages. Done for launch with sender-only unread delete, attachment cleanup, realtime refresh on message deletes, and delivered/read indicators based on the other member's read marker.
- Booking flow: verified tattoo artists and studios should get a TimeTix-style booking system inside TheTattooCore: artists/studios can define appointment types, consultation/tattoo time slots, working days, blackout dates, buffer times, booking windows, cancellation/reschedule rules, required or optional deposits, and shop/artist-specific availability. The booking system should connect to Google Calendar, Apple/iCloud Calendar, or standard iCalendar feeds where supported, show bookable availability on profiles and in DMs, create appointment requests from profiles/DMs, and collect tattoo deposits through the shared TTC payment gateway before an appointment is confirmed. TTC should record and display a small transparent processing/platform fee on booking deposits, then route deposit checkout through the platform so artists do not need to build separate payment flows. Started with RLS-protected `booking_requests`, verified artist/studio recipient checks, profile-side booking request forms, requested deposit amount capture, transparent TTC processing fee recording, booking request notifications, Account-side incoming/sent booking cards with artist accept/decline controls, optional scheduled appointment times on accepted requests, final artist/studio deposit adjustment before acceptance opens checkout, private participant-only `.ics` calendar downloads for scheduled bookings, guarded Stripe Checkout for accepted deposits only, client cancellation for unpaid requests before checkout starts, artist/studio cancellation for unpaid accepted bookings, public cancellation policy notes on booking-enabled profiles, DM-thread surfacing for attached booking requests, manual booking availability settings on verified artist/studio profiles, safe public booking-link/calendar-note fields so artists can point members to existing booking or calendar request pages during launch, slot-template tables, appointment-type setup, blackout dates, public profile appointment/slot previews, and booking request references to selected appointment types or preferred weekly slots. Next steps are true Google/iCloud/iCalendar account connection or feed sync, availability conflict checks, deeper paid-refund handling, and production booking/payment review.
- DMs need a focused test pass with the regular test user and moderator test user before app submission. Mobile inbox/thread routing, read-state clearing, signed attachment display, direct thread read markers, notification deep-link routing, realtime refresh subscriptions for message/photo inserts, invalid-thread fallback handling, and explicit sender membership checks are improved for launch. The read-state refresh loop is fixed by only updating `last_read_at` when a selected thread has unread/incoming messages, realtime attachment refresh noise is reduced to the active thread, and the browser Supabase client now has stable public config for deployed realtime use. Real send/receive testing still needs logged-in test sessions.
- Notifications: in-app alerts and unread badges first, important email second, installed app push third, then mobile app push for iOS and Android apps. Started for launch with safe push display/click routing, signed-in subscription storage, a key-gated Notifications-page control, and quiet-hour/preference helpers that future push/email delivery can respect.
- Composer: one floating plus button that opens the correct posting form for the current column.
- Main column swiping should snap one column at a time. A horizontal swipe should land on the next column and stop instead of sliding quickly across multiple sections. Done for launch with a controlled mobile snap rail.
- Column swipes should also reset the newly active column to the top and refresh against that column's newest ranked posts so a user lands at the current top of 4U, Gossip, Stuff, Gigs, Merch, or DM instead of staying mid-scroll from the previous column. Done for launch with top reset on swipe/tab/hash changes plus a cooldown-protected route refresh when the active column changes so server-ranked data can refresh without hammering the app.
- Column order for launch planning is 4U, Gossip, Stuff, Gigs, Merch, then DM. Merch is now in test-mode checkout with Stripe; keep it gated until seller approval, tax, shipping, refunds, fulfillment, payout, and production payment rules are ready.
- Feed ranking should become a user-specific recommendation algorithm across every column: prioritize followed artists/studios/vendors, recent posts from people the member interacts with, similar tattoo styles/categories/keywords to posts they like/save/comment on, coarse location/language relevance, and freshness. Keep the algorithm understandable, non-AI-branded, safety-filtered, and column-specific: 4U ranks image/reel work, Gossip ranks discussions, Stuff ranks verified professional listings, Gigs ranks opportunities, and Merch ranks safe fan-buyable products. Started for launch with a lightweight home-column score using accepted follows, saved/liked items, style/category overlap, location personalization, engagement, and recency after safety filters.
- Ranking must preserve safety and moderation rules before personalization: hidden, removed, blocked-user, sensitive logged-out, private, unverified restricted-commerce, or unsafe content should be filtered before scoring. Freshness should matter enough that users still see new work, not only historical favorites.
- For launch performance, keep ranking simple and database-friendly before building a heavier recommender: combine follow affinity, interaction affinity, style/category overlap, location/language match, engagement velocity, and recency into a score, then paginate 25 at a time.
- Long feeds and queues should load 25 items at a time with a clear Load more button for each next batch of 25. Apply this to 4U, Gossip, Stuff, Gigs, Merch, notifications, saved items, search results, comments/replies, admin queues, account ad history, account order history, and future long lists so the site stays fast as volume grows. Done for launch on the home 4U, Gossip, Stuff, Gigs, Merch, notifications, saved items, search results, DM inbox, account ad history, account Merch order history, and 4U/Gossip detail comments, with follower/following and dedicated admin queues using 50-item page navigation.
- Main feed cards should not render full comments inline. Home 4U/Gossip cards should show only the visible comment count; clicking the count opens the detail page where comments, replies, edit/delete/report controls, and Load more live. Done for launch, including reduced home-feed comment select payloads.
- Admin queues should get per-tab Load more controls after the simple 25-item caps are stable, so verification, reports, users, ads, data requests, media review, and activity logs can scale without returning to one giant admin page. Dedicated admin queue pages now use 50-item page navigation where real queues exist.
- Admin sections must become dedicated pages as volume grows, especially Users, Ads, Payments, Stuff, Merch, Reports, Verification, Data Requests, Content, Gigs, Media Ops, and Mail Settings. Each long admin list should use real pagination instead of an endless dashboard section; Users, Ads, Payments, Data Requests, Verification, Reports, Content, Stuff, Gigs, Mail Settings, Media Ops, and Merch have dedicated page layers with shared section navigation so admins can move between queues without overloading the overview. Admin Payments now paginates Stripe webhook receipts 50 at a time, uses exact status counts instead of loading sampled rows into the page, and shows an ops watch for stale pending Merch checkouts, stale booking deposit checkouts, or active ads with unpaid/problem payment; those warnings now link into filtered Admin Merch/Admin Ads queues where dedicated filters exist. Admin Ads supports campaign/payment filters, and Admin Merch supports product/order filters so long queues stay reviewable.
- Admin `/admin` must stay an overview-only command center: high-level counts, health/status cards, and links into dedicated admin pages. It should not contain full user, report, verification, ad, payment, Stuff, Merch, Gig, content, mail, or data-request queues because those pages will get long over time. Done for launch with short overview cards, recent activity, and dedicated queue links.
- Owner/admin support tooling should include a controlled way to create tester/member accounts from Admin for QA, demos, dummy marketplace data, and support workflows. Done for launch on Admin > Users with owner-only access, server-only Supabase admin auth, confirmed tester creation, and audit logging.
- Search and Saved should mention Merch as limited/checkout-in-testing until products, seller approval, tax, shipping, refunds, fulfillment, payouts, and production payment rules are ready. Search now includes active, indexable public Merch previews, and Merch products can be saved from home/detail views and rendered in Saved.
- Mobile composer: posting cards must fit mobile browser screens, scroll to the publish button, and keep advanced fields behind expandable sections. Done for launch.
- Composer uploads should not show a sensitive-content option during launch; visible nudity is not allowed, so members must crop or cover private areas before posting.
- Profiles need avatar/profile-photo upload with client-side image optimization, public display on profile pages, and rollout into feeds, comments, DMs, search, notifications, and detail pages. Done for launch, including 4U/Gossip detail comments.

Done for launch with editable banner upload/removal, public profile cover rendering, private-profile hiding, profile share-card preference for public banners, and banner-aware discovery surfaces in Search, Saved, followers, and following.
- Profiles need a small public bio plus safe outbound links for website and common social accounts so artists, studios, vendors, and enthusiasts can point visitors to booking, portfolios, and brand channels. Done for launch with editable bio, website, Instagram, TikTok, Facebook, YouTube, and X links rendered on public profiles when the profile is viewable, bio-aware profile search previews, shared URL sanitizing, user-generated outbound link trust flags, an artist-to-studio/shop profile link, and a public linked-artist list on studio/shop profiles.

Done for launch with a labeled Flag + Report trigger, inline report form, and shared report/moderation coverage for 4U, Gossip, Stuff, Gigs, Merch, profiles, and comments.
- Mobile report menus must stay inside the viewport; three-dot report cards in 4U, Gossip, Stuff, Gigs, and profiles should not open off-screen on narrow browsers. Done for launch by using an inline report panel.
- Report triggers should be labeled enough to reduce false taps; icon-only menus are acceptable only when the surrounding UI makes the report purpose unmistakable. Done for launch with a Flag + Report trigger.
- Comments and DMs: add richer expression over time, including emoji, GIFs, photo attachments, and simple media attachments in direct messages. DMs have emoji shortcuts plus private JPG/PNG/WebP/GIF photo attachments with lightbox viewing for launch; 4U and Gossip detail comments/replies now support optional JPG/PNG/WebP/GIF attachments up to 5 MB with lightbox viewing, while main feeds still keep comments collapsed to counts for speed.

## Platform Stance

- TheTattooCore should openly emphasize that it does not use AI-generated art, AI search, AI feeds, or AI-driven creator replacement features.
- Keep the platform centered on real body-art artists, studios, collectors, vendors, and enthusiasts.
- Stand against corporate takeover pressure in the tattoo and body-art industry; keep the community focused on independent artists, studios, craft, culture, and ethical vendors.
- Do not promote or normalize unprofessional tattooing, unlicensed studios, or unsafe "scratcher" activity.
- Marketplace rules should prevent sales of tattoo equipment to unlicensed or unprofessional buyers where the platform can reasonably enforce it.
- Stuff access rules should allow public/fan browsing but restrict buy, sell, trade, and seller-contact actions to verified artists, studios, and vendors. Done for launch.
- Stuff UI copy should explicitly call out tattoo machines, needles, pigments, tubes, and professional shop gear as verified-only activity so fans understand browse-only access. Done for launch.
- Merch rules should allow public fan purchases of safe brand goods while still requiring seller approval for artists, studios, vendors, and official TheTattooCore store inventory. Admin product approval/activation now checks seller license verification for non-official Merch before it can go public, and public Merch discovery/detail/checkout/sitemap exposure hide or block non-official products if seller approval lapses. Merch products now also have shared moderation status, public reads and checkout require active moderation, reports can be moderated, and `/admin/content` includes a Merch review tab.
- Merch must stay separate from professional equipment, regulated services, unsafe products, counterfeit goods, adult sexual products, and anything that would undermine app-store or payment-provider safety.
- Merch payments use Stripe Checkout and webhook-backed order status in test mode. Test checkout, checkout inventory reservations, paid webhooks, Stripe event-ID dedupe, webhook retry-safe paid-order transitions, atomic paid-order inventory decrement, failed/expired checkout reservation release, buyer in-app/email failed-payment and cancelled-checkout alerts, admin and seller available/reserved inventory visibility, status-aware checkout return copy, buyer/seller/admin order history with subtotal, TTC fee, shipping, tax, discount, and total breakdowns, buyer-scoped receipt lookup with item lines, printable receipt polish, pending-order cleanup when checkout or pre-checkout item setup fails, safer local-order-before-Stripe-session creation, guarded Stripe-session attach only to still-pending unassigned orders, product refreshes after fulfillment/refund updates, shipping-required checkout behavior, admin order item visibility with lifecycle dates, basic admin order controls, retry-safe paid/refund notifications, seller in-app paid-sale alerts, seller-side paid line-item fulfillment with optional carrier/tracking capture, buyer in-app fulfillment alerts, buyer in-app refund alerts, and audit-only dispute/chargeback logging for linked Merch payments are wired. Next payment work is production seller approval, Stripe Connect or manual payout policy, deeper refund handling, tax/shipping-rate workflow, dispute procedures, and payment review.
- Stripe should become the site-wide payment gateway/POS layer for ads, Merch, and any future paid Stuff/booking/payment flows, so buyers, sellers, vendors, advertisers, and future appointment-deposit payers can pay or receive payouts through the platform without each area inventing separate payment logic. Started with Merch checkout as the first gateway-backed flow, ad checkout as prepaid campaign funding, booking deposits using the shared platform-fee structure with admin payment-state visibility, and audit-only dispute/chargeback logging for linked Merch, ad, and booking payments.
- Seller and vendor onboarding should eventually collect the payout details Stripe requires through a secure Stripe-hosted flow, not by storing bank/debit-card payout data directly in TheTattooCore. Buyers should pay through Stripe Checkout or a Stripe-hosted payment element, and TTC should avoid handling raw card or bank credentials.
- TTC platform fees should be transparent and configurable by payment type. Start with a small 1-2% platform fee concept for Merch/vendor/artist sales, booking deposits, and ads where legally and payment-rule appropriate, record it on every order/payment, and decide per flow whether the fee is buyer-paid, seller-deducted from payout, or split. Started with a shared 2% test-mode fee helper used by Merch, ad checkout, and booking request deposit records, recorded as `platform_fee_cents`, and displayed to buyers/admins where checkout exists. Taxes, refunds, disputes, card processing fees, and app-store rules need review before production rates are final.
- Support freedom of body-art expression and avoid unnecessary censorship when content follows safety, consent, legality, and adult-content guidelines.
- Draw firm safety lines around pornography, sexualized content, exploitation, minors, harassment, scams, unsafe practices, and content showing unprofessional or potentially deadly harm.
- Position the community as a safe space for body-art culture to express itself without unwanted AI, spam, or unprofessional practice watering it down.
- Blocking should hide profile content from the blocker, remove follow relationships, and prevent new DMs in either direction. Done for launch for profile blocks and comment-owner blocks from 4U/Gossip moderation controls.
- Launch trust surfaces should repeat these values in user-facing places, including `/login`, `/terms`, `/privacy`, account verification, account advertising settings, share metadata, and install metadata. Done for launch, including front-door no-AI, no-scratchers, verified Stuff, and independent-community positioning.
